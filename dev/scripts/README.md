# Developer Scripts

This directory contains automation utilities designed to bridge the gap between low-level assembly source code and the VBA environment. These tools are essential for maintaining the Wasabi reactive engine.

> [!IMPORTANT]
> These scripts are intended for development and maintenance purposes only. The resulting opcodes generated by these tools are hardcoded into the production modules to ensure the end-user has no external dependencies.

## Script Index

### 1. compile_asm.bat
A Windows batch utility that automates the assembly process.
* **Purpose**: Compiles assembly source files (.asm) into raw machine code binaries (.bin).
* **Source Path**: `../asm/safe_thunk_x64.asm` and `../asm/safe_thunk_x86.asm`.
* **Output Path**: `../asm/safe_thunk_x64.bin` and `../asm/safe_thunk_x86.bin`.

> [!NOTE]
> This script requires NASM (Netwide Assembler) to be installed and accessible via the system PATH environment variable.

### 2. bin_to_vba.py
A Python-based extraction tool for binary-to-VBA conversion.
* **Purpose**: Reads compiled .bin files and outputs formatted hexadecimal byte arrays.
* **Functionality**: Generates the exact `asm(i) = &Hxx` sequences required for the initialization block.

## Workflow

> [!TIP]
> Always run the compilation script before the conversion script to ensure you are working with the most recent version of the assembly logic.

1. **Modify Source**: Update the assembly logic in `dev/asm/`.
2. **Compile**: Execute `compile_asm.bat` to refresh the binary blobs.
3. **Convert**: Run `python bin_to_vba.py` to generate the new opcode strings.
4. **Update VBA**: Copy the hexadecimal output into the appropriate conditional compilation blocks within the `Wasabi` module.

## Technical Safety

> [!CAUTION]
> Incorrect modifications to the assembly source or improper injection of opcodes can lead to immediate memory corruption and host application crashes. Ensure that the binary size matches the expected architecture (42 bytes for x64, 21 bytes for x86) before updating the VBA source.

### Data Execution Prevention (DEP)
The Wasabi framework must allocate memory using specific WinAPI constants to remain compatible with DEP. The scripts in this directory assist in preparing the code that will eventually be placed into memory segments marked with `PAGE_EXECUTE_READWRITE`.
